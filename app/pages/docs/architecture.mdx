## Contract Architecture

### Hub-Based Routing

All game contracts extend the **Railway** framework's Hub pattern. Players are grouped into **Railcars** (groups), and each hub (Lobby, Round 1-4, Winners) processes the railcar as it moves through the game flow:

```
Lobby вҶ’ Round 1 вҶ’ Round 2 вҶ’ Round 3 вҶ’ Round 4 вҶ’ Winners
```

The `HubRegistry` maps hub aliases (e.g., `hjivemind.round1`) to contract addresses, enabling contract-to-contract discovery.

### State Flow

```
Player joins Lobby
  вҶ’ ScoreKeeper.setGameID()
  вҶ’ Railcar created, player added
  вҶ’ Countdown starts (or starts immediately if limit reached)

Lobby starts game
  вҶ’ Railcar moves to Round 1
  вҶ’ HjivemindKeeper queues SeedRound action

AutoLoop delivers VRF seed
  вҶ’ Round selects question via seed % questionCount
  вҶ’ Question phase begins

Players submit hashed answers
  вҶ’ Phase advances to Reveal (when all submitted or time expires)

Players reveal answers
  вҶ’ Scores calculated (submission + winning + fast reveal bonus)
  вҶ’ Phase advances to Completed
  вҶ’ Railcar moves to next round (or Winners)

Winners phase
  вҶ’ Final rankings calculated
  вҶ’ Prize pool distributed to top 4
```

## AutoLoop Integration

[AutoLoop](https://tryautoloop.com) provides two critical services:

### 1. Keeper Automation

AutoLoop workers continuously call `shouldProgressLoop()` on the HjivemindKeeper. When work is pending, they call `progressLoop()` to advance game state. This handles:

- Starting games when lobby countdown expires
- Advancing round phases when time limits are reached
- Moving railcars between rounds
- Triggering winner calculations

### 2. Verifiable Randomness (VRF)

Each round needs a random seed to select questions. AutoLoop workers generate **ECVRF proofs** (Elliptic Curve Verifiable Random Function) that are verified on-chain:

1. Worker detects `SeedRound` action in keeper queue
2. Worker computes deterministic seed: `keccak256(keeperAddress, gameID)`
3. Worker generates ECVRF proof using its secp256k1 private key
4. Worker submits proof in `progressLoop()` call
5. `VRFVerifier` contract verifies the proof on-chain (~3k gas via ecrecover)
6. Verified randomness delivered to GameRound

The VRF envelope format:

```solidity
abi.encode(
    uint8 vrfVersion,          // 1 = ECVRF-SECP256K1-SHA256-TAI
    uint256[4] proof,          // [gamma_x, gamma_y, c, s]
    uint256[2] uPoint,         // precomputed U point
    uint256[4] vComponents,    // precomputed V components
    bytes gameData             // original queue data
)
```

## x402 Payment Architecture

The game uses a **two-chain model** for payments:

| Chain | Role |
|-------|------|
| Base Sepolia | USDC micropayments (x402 protocol) |
| Ethereum Sepolia | Game smart contracts |

The **relayer** bridges these chains: it verifies USDC payment on Base Sepolia, then submits Sepolia transactions using a funded server-side wallet. This means players never need Sepolia ETH.

See [x402 Payments](/docs/x402) for implementation details.

## Security Model

### Commit-Reveal

Answers are hashed before submission to prevent front-running and copying:

```
hash = keccak256(abi.encode(questionAnswer, crowdAnswer, secretPhrase))
```

The secret phrase adds entropy вҖ” without it, someone could brute-force the hash since there are only 16 possible answer combinations (4 choices x 4 crowd predictions).

### Access Control

Every contract uses OpenZeppelin `AccessControlEnumerable` with role-based permissions. No contract function that modifies state is publicly callable without the appropriate role.

### VRF Soundness

The ECVRF implementation ensures question selection is:

- **Unpredictable**: Workers cannot choose which question is selected
- **Verifiable**: Anyone can verify the proof was correctly computed
- **Deterministic**: Same seed always produces the same output
